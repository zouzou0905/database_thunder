# 后续优化清单

本文记录当前系统可正常使用前提下的后续优化项。以下内容不是立即修复任务，不应在没有明确业务触发条件时主动改动数据库结构、查询路径或索引。

## 数据库与查询路径

| 优先级 | 优化项 | 当前状态 | 触发条件 | 备注 |
| --- | --- | --- | --- | --- |
| P1 | 统一横向对比数据源选择逻辑 | `list/count/export` 各自判断 snapshot、interval、scenario、range cache、live CTE | 预览条数或导出明显变慢，或新增更多固定区间 | 先抽象 source resolver，再让三个入口复用 |
| P1 | 横向对比 count 支持 interval snapshot | 列表支持 interval snapshot，count 路径相对不完整 | 数据导出“预览条数”超过 5-10 秒 | 避免预览条数绕到 range cache/live CTE |
| P1 | 横向对比导出使用专用 SQL | 当前导出按 500 行分页复用列表查询 | 导出 2 万条以上明显慢 | 避免重复 count、months 查询和标签补全 |
| P2 | 将 `keyword_compare_range_cache` 降级为兜底路径 | 仍存在自动 `_ensure_range_cache()` 逻辑 | 非常规区间查询频繁触发冷计算 | 不建议直接删除；先观察是否仍有业务价值 |
| P2 | 审计大索引使用率 | 多个大索引 `idx_scan` 很低或为 0 | 快照刷新变慢、磁盘增长过快、导入变慢 | 删除索引前需再次采样 `pg_stat_user_indexes` |
| P3 | 重构机会池 where 构造 | 当前通过字符串 `replace()` 给字段加别名 | 机会池筛选逻辑继续增加 | 改成构造条件时直接传入 alias |

## 观察指标

建议只在出现以下现象时启动优化：

- 数据导出预览条数耗时超过 5-10 秒。
- 横向对比导出 20,000 条以上明显卡顿。
- PostgreSQL 磁盘占用持续异常增长。
- 快照刷新耗时明显上升。
- 非常规日期区间查询频繁触发 30 秒以上等待。
- 新增多个节日、促销季或固定区间快照后，source 判断逻辑变得难以维护。

## 当前决策

当前系统可以正常使用，以上项目暂不执行。后续如需优化，优先从 P1 项开始，并避免同时修改查询逻辑和数据库索引。
