# 02-数据模型与计算链路

## 数据库分层

当前 PostgreSQL 数据库按四层理解：

```text
导入管理层
原始与标准化层
分析计算层
运营动作层
```

## 关键表

### 导入管理层

```text
import_batches   每次导入任务
source_files     每个 CSV 文件，含文件 hash，避免重复导入
```

### 原始与标准化层

```text
raw_aba_rows             CSV 原始行
keywords                 关键词主表
keyword_monthly_metrics  关键词月度指标
```

`keyword_monthly_metrics` 是后续趋势、机会池、横向对比的核心底表。

### 分析计算层

```text
keyword_monthly_trends              月度趋势结果
keyword_ops_monthly                 运营宽表
keyword_selection_candidates_monthly 关键词机会池缓存表
keyword_compare_snapshot_monthly    横向对比快照表
v_mb_product_selection_candidates    实时候选视图，缓存缺失时兜底
```

当前前端机会池主查询来源是：

```text
keyword_selection_candidates_monthly
```

实时视图只用于兜底，不应成为多人高频访问的常态。

### 运营动作层

```text
users                         用户
keyword_user_actions          收藏、状态、备注等运营动作
keyword_selection_exclusions  禁用词、品牌词、排除规则
```

## 月份口径

系统中的“本月”不是现实日期，而是运营选择的分析月份。

例如分析月份为 `2026-04`：

```text
本月 = 2026-04
上月 = 2026-03
近 3 个月 = 2026-01 到 2026-03
去年同期 = 2025-04
```

前端可以传 `YYYY-MM`，后端会统一转换为 `YYYY-MM-01`。

## 计算链路

```text
CSV 文件
  -> import_csv.py
  -> raw_aba_rows
  -> keywords
  -> keyword_monthly_metrics
  -> calculate_trends.py
  -> keyword_monthly_trends
  -> keyword_ops_monthly
  -> keyword_selection_candidates_monthly
  -> keyword_compare_snapshot_monthly
```

`calculate_trends.py` 默认刷新关键词机会缓存和横向对比快照。只有明确需要跳过大表写入时，才使用 `--skip-cache-refresh`。

## 横向对比

横向对比按同一个 `keyword_id` 比较多个已导入月份：

```text
搜索量变化
排名变化
出现月数
连续出现情况
增长率
数据源历史排名参考
```

数据源原始字段中的“上月排名、四个月前排名、十二个月前排名”不会和系统横向对比冲突。它们是数据源参考值，系统横向对比是基于数据库已导入月份重新计算的结果。

## 性能要求

```text
常用筛选字段必须有索引
后端必须分页
单次接口不能返回全量数据
活跃月份必须提前生成缓存
缓存缺失时允许兜底，但页面必须提示
```
